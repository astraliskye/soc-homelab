# T1059.001 - Powershell

Powershell downloads Mimikatz and dumps credentials.

## Hypothesis

Running T1059.001 test number 1 should produce a event 4104 script block event downloading Mimikatz. There should also be an event 10 process access against LSASS.exe.

## Attack

Run Atomic Red Team T1059.001 test number 1

```powershell
Invoke-AtomicTest T1059.001 -TestNumbers 1
```

![Atomic Red Team invocation](../screenshots/T1059.001/atomic-red-team-execution.png)

## Telemetry

### Powershell script block (Event ID 4104)

Powershell script block access logs decoded script blocks when they are executed. There are some legitimate uses of powershell, particularly by IT administrators. When looking to detect malicious powershell use, it is useful to look up the `user_id` of the event and inspect the role of the user executing the powershell command. It is also useful to select for known suspicious substrings such as `Invoke-Mimikatz`.

![Powershell script block event](../screenshots/T1059.001/powershell-script-block-detection.png)

### Sysmon Process Access (Event ID 10)

Sysmon is configured to log process access including the target process and the accessing process. For this attack chain, the process access event will fire regardless of how the attacker got Mimikatz on the system. There are legitimate cases where LSASS can be accessed, such as by an EDR or anti-virus. Since the legitimate use cases are few, the analyst should write detection rules for LSASS accesses by any process save for the particular EDR or AV their enterprise may be using.

![LSASS process access event](../screenshots/T1059.001/lsass-process-access-detection.png)

## Detection

### Detecting LSASS access with credential-related GrantedAccess

```spl
index="main" source="XmlWinEventLog:*" EventID="10" TargetImage="*lsass.exe" GrantedAccess IN ("0x1010", "0x1f1fff", "0x1ffff", "0x1438", "0x143a", "0x0810", "0x0820", "0x0800") | stats count by host, SourceImage, SourceProcessGUID, GrantedAccess, CallTrace
```

This will detect successful potential credential-related privileged accesses to LSASS.exe process memory. False positives may come from legitimate access by EDR or AV. During testing, Mimikatz was only able to obtain 0x1000 access, so it was not picked up by this specific query. I added the condition for the GrantedAccess to tighten the scope of the query to reduce possible false positives. GrantedAccess values were pulled from the [Sysmon community guide](https://github.com/trustedsec/SysmonCommunityGuide/blob/master/chapters/process-access.md#understanding-access-masks)

### Detecting Mimikatz related Powershell script blocks

```spl
index="main" source="XmlWinEventLog:*" EventID="4104" (ScriptBlockText="*sekurlsa::*" OR ScriptBlockText="*Invoke-Mimikatz*" OR ScriptBlockText="*kerberos::*") | stats count by host, User, ScriptBlockText
```

This detects specific strings related to Mimikatz credential dumping. Rare occurrences of false positives.

## Investigating

1. Check SourceImage to see if it makes sense
2. Check SourceProcessGUID
3. Check what happened on the host over the last 15 minutes or so
4. Read through the CallTrace

## Notes

Mimikatz fails to dump credentials, citing `ERROR kuhl_m_sekurlsa_acquireLSA ; Handle on memory (0x00000005)`. This is likely due to LSA protecting the LSASS process memory and only granting 0x1000 access (PROCESS_QUERY_LIMITED_INFORMATION). It can be circumvented by loading a driver and disabling protection for the LSASS process. The primary detection query will only match on successful granting of access, so failures like this will not be caught, though they may indicate malicious activity on the network.
